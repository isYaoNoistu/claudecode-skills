# Claude Skills 白皮书总结文档
## 核心定位
本学习文档提炼白皮书**核心知识点、实操要点、易混概念、落地避坑**，聚焦**快速理解+立即落地**，剔除冗余案例和背景，适配你的**OpenCode+运维Skills**开发场景，10分钟可完成核心学习。

## 一、核心速览
### 1. 一句话定义
Skills 是**模块化的AI能力包**，由`SKILL.md`（核心）+ 可选脚本/模板/参考文档组成，让AI（Claude/OpenCode）按需加载、复用标准化工作流程，本质是给AI的**领域工作手册**。
### 2. 解决的3个核心问题
- 重复工作低效：无需每次对话重新描述流程（如运维的Prometheus查询、网页检测步骤）；
- Token 严重浪费：传统Prompt/MCP全量加载，Skills**按需分阶段加载**，大幅降低Token消耗；
- 专业知识难复用：运维、研发等领域的标准化流程可结构化存储、团队共享、版本管理。
### 3. 核心价值
- 知识复用：运维的监控查询、网页检测、故障排查流程**一次编写，AI自动调用**；
- 低成本适配：仅需写Markdown，无需后端开发能力，比MCP门槛低；
- 团队协同：运维规范可做成Skills仓库，团队统一执行标准，避免人为操作差异；
- 适配OpenCode：OpenCode兼容Claude Skills规范，你编写的运维Skills可直接复用该标准。

## 二、核心架构：三层加载机制Skills的核心设计
这是Skills最核心的技术点，也是**Token效率高的关键**，采用**渐进式披露**，分阶段加载，用多少加载多少，完全适配AI处理运维需求的场景（简单需求轻量加载，复杂需求深度加载）。

| 加载层级 | 核心内容 | 加载时机 | Token成本 | 核心作用（运维场景） |
|----------|----------|----------|-----------|----------------------|
| Level 1 元数据 | `SKILL.md`开头的YAML配置（name/description/metadata） | AI启动时**永久加载** | ~100 tokens/个 | 让AI知道“有这个运维Skill、什么时候用”（如含“Prometheus、监控指标、网页查询”关键词时触发） |
| Level 2 指令 | `SKILL.md`的Markdown主体（功能说明、使用场景、执行步骤） | 匹配到description关键词**触发时加载** | <5000 tokens | 给AI提供运维操作的**标准化步骤**（如Prometheus查询的参数、网页检测的各阶段流程） |
| Level 3 资源 | 脚本（.py/.sh）、模板、参考文档（如运维排障手册） | 指令中**引用时才加载** | 几乎无限制（仅脚本输出占Token） | 提供运维的**可执行脚本、标准化模板**（如PromQL查询脚本、网页检测的输出表格模板） |

### 关键结论
对运维Skills而言，**把核心流程写在Level2，把可执行的查询/检测脚本放在Level3**，既保证AI能按规范执行，又能控制Token消耗。

## 三、最简实操：从0创建一个Skill
### 1. 最小可行Skill（MVS）结构（仅需1个文件）
无需复杂目录，核心仅需`SKILL.md`，满足OpenCode/Claude的加载要求，运维轻量Skill可直接用此结构：
```markdown
---
# Level1 元数据（必需，严格YAML格式）
name: 运维技能名（小写+连字符，如prometheus-query、webpage-check）
description: 核心功能+触发场景+关键词（必写，AI匹配的关键，如“查询Prometheus监控指标，分析性能趋势，使用场景：查询CPU/内存/磁盘指标、编写PromQL、排查监控异常”）
license: MIT
compatibility: opencode
metadata:
  audience: 运维工程师
  workflow: 监控/网页检测/故障排查
---
# Level2 指令（核心，Markdown格式）
## 功能说明
简洁说明该Skill能做的运维操作（如“调用Prometheus API查询指标，分析趋势，调试PromQL”）
## 使用场景
明确AI触发的具体运维场景（如“需要查询Prometheus监控数据、分析系统性能瓶颈、编写/优化PromQL语句时使用”）
## 执行步骤
标准化的运维操作步骤（如Prometheus查询：1. 确认服务地址；2. 执行查询脚本；3. 分析返回数据；4. 输出结构化结果）
## 输出格式
运维结果的标准化格式（如表格、关键指标清单，贴合你的使用需求）
```

### 2. 完整Skill目录结构（复杂运维Skill用，如含多脚本/模板）
按白皮书规范，可扩展为以下结构，OpenCode能自动识别，适合团队共享的运维Skills仓库：
```
skill-name/  # 与元数据的name一致
├── SKILL.md  # 核心文件（必需）
├── scripts/  # 运维可执行脚本（可选，如promql_query.py、web_check.sh）
├── templates/  # 输出模板（可选，如监控报告模板、网页检测表格模板）
└── references/  # 参考文档（可选，如PromQL手册、运维排障指南）
```

### 3. SKILL.md编写黄金规则（重点：description）
`description`是**AI能否正确触发Skill的关键**，白皮书反复强调，针对运维场景的编写要点：
✅ 正确写法：**核心功能+次要功能+触发关键词+使用场景**
```yaml
description: 查询和分析Prometheus监控指标，编写调试PromQL语句，识别资源瓶颈，使用场景：查询服务器CPU/内存/磁盘指标、分析性能趋势、排查监控异常，关键词：Prometheus、监控指标、PromQL、性能分析
```
❌ 错误写法：太简短/无场景（如“Prometheus查询”）

## 四、易混概念速记：Skills vs 其他方案
你开发运维助手时会接触Prompt/MCP，白皮书核心对比了三者差异，**重点掌握Skills与MCP的互补关系**（你目前不用MCP，仅需做区分，后续扩展可复用）。

### 1. Skills vs Prompts（传统提词）
| 维度 | Prompts | Skills |
|------|---------|--------|
| 复用性 | 单次对话，每次重写 | 一次创建，全局/团队复用 |
| 加载方式 | 手动输入，全量加载 | 自动匹配，按需加载 |
| 适用场景 | 一次性运维操作 | 标准化重复运维流程（如监控查询、网页检测） |

### 2. Skills vs MCP（核心，互补而非竞争）
这是白皮书的重点，**你的运维助手目前不用MCP，仅需记住：MCP是“数据源/工具桥”，Skills是“使用手册”**
| 维度 | MCP | Skills |
|------|---------|--------|
| 核心定位 | 连接外部系统的**桥梁**（如SSH、数据库、监控平台的实时数据） | 使用外部能力的**标准化手册**（如怎么用Prometheus数据做分析、怎么用SSH执行排障命令） |
| 技术门槛 | 需后端开发，配置服务器/JSON | 仅需写Markdown，会写运维文档即可 |
| 数据类型 | 实时外部数据 | 静态运维规范+可执行脚本 |
| 适用场景 | 需访问**实时外部系统**（如直接操作目标服务器、查询实时业务数据） | 需**标准化工作流程**（如监控分析、网页检测、故障排查的固定步骤） |
| 关系 | 互补：MCP提供“运维数据/工具”，Skills定义“怎么用这些数据/工具做运维” |

### 3. 核心结论
- 目前你不用MCP：**专注写Skills即可**，基于OpenCode部署机的本地命令（curl/promtool/nc）实现运维操作；
- 后续需跨机器操作（如直接SSH目标服务器）：**加MCP做数据/工具桥，Skills定义运维流程**，两者组合是最优解。

## 五、落地最佳实践（运维场景适配，直接复用）
白皮书给出了大量通用最佳实践，提炼**适配运维Skills开发**的核心要点，让你的Skills更易触发、更易维护、更适配OpenCode：
### 1. 单一职责：一个Skill管一个运维场景
❌ 不要做“全能运维Skill”（含监控、网页、SSH、日志所有功能）；
✅ 拆分细粒度Skill（如`prometheus-query`/`webpage-check`/`log-analysis`），AI可组合调用，也便于维护。
### 2. 脚本优先：运维可执行操作写脚本，而非让AI生成代码
- 原因：脚本**预先测试、确定性强、不占Token**（仅输出占Token），AI生成代码易出错、占Token；
- 实操：把PromQL查询、网页检测各阶段耗时统计、日志过滤等操作写成`scripts/`下的py/sh脚本，Skills中仅写“执行XX脚本”的步骤。
### 3. 渐进式披露：复杂运维知识拆分成参考文档
- 核心：把**基础运维流程**写在`SKILL.md`（Level2），把**复杂排障手册、PromQL高级语法**放在`references/`（Level3），AI仅在需要时加载；
- 实操：如网页检测Skill，基础的DNS/TCP/TLS检测步骤写在Level2，复杂的HTTPS证书排障、超时问题根因分析放在`references/排障指南.md`。
### 4. 输出标准化：贴合运维的阅读习惯
Skills中明确输出格式（如表格、关键指标清单、异常+建议），让AI返回的结果直接可用，无需二次整理（如你之前的Postman风格网页检测，明确各阶段耗时的表格格式）。

## 六、安全&性能避坑（必看，运维场景重点）
白皮书花大量篇幅讲局限性和注意事项，提炼**运维助手开发中最易踩的坑**，重点关注**代码执行安全**和**Token性能优化**：
### 1. 安全风险（运维服务器核心风险，必须规避）
Skills可执行脚本，若脚本存在恶意代码/漏洞，会影响OpenCode部署机，甚至泄露运维服务器信息：
✅ 防护措施：

- 仅用**自己编写/审计过**的Skills，不使用未知第三方Skills；
- 审查所有脚本：禁止脚本中出现`rm -rf`/`eval`/`exec`等高危命令，禁止硬编码API密钥/密码；
- 最小权限：运行OpenCode的账号仅赋予**运维必要权限**（如curl/promtool/nc），不使用root直接运行。
### 2. 性能优化（控制Token消耗，提升AI响应速度）
✅ 核心优化点：
- 精简元数据：`description`简洁明了，不写冗余内容；
- 拆分大Skill：细粒度拆分运维场景，避免单个Skill加载过多内容；
- 脚本隔离：把大量代码逻辑放在Level3的脚本中，不占用Level2的Token。

## 七、白皮书核心结论
1. 你目前的**OpenCode+无MCP+运维Skills**方案完全可行，贴合Skills的核心使用场景（标准化运维流程）；
2. 按Claude Skills白皮书的规范编写运维Skills（尤其是`SKILL.md`的格式和description），可保证**兼容性、可维护性、可扩展性**（后续加MCP/团队共享可直接复用）；
3. 运维Skills的核心是**把你的标准化运维操作写成AI能理解的手册+可执行脚本**，让AI替代人工执行重复的监控查询、网页检测、故障排查步骤。

## 快速学习落地清单

3. 验证：在OpenCode中输入运维需求（如“查询Prometheus CPU指标”），看AI是否能正确匹配对应的Skill；
4. （可选）把常用的PromQL查询、网页检测命令写成脚本，放在`scripts/`目录，让AI直接执行。
